Database Schema (V1)

This reflects everything agreed so far: employee registrations (split into identity + per-trade registration), company requests (split into identity + per-need request), the shared job_categories lookup, and a plain admin login. Built so that V2 (employee/company login, matching, payroll) only adds tables and columns — nothing here needs to be restructured later.

erDiagram
    JOB_CATEGORIES ||--o{ EMPLOYEE_REGISTRATIONS : "categorizes"
    JOB_CATEGORIES ||--o{ COMPANY_REQUESTS : "categorizes"
    EMPLOYEES ||--o{ EMPLOYEE_REGISTRATIONS : "has"
    COMPANIES ||--o{ COMPANY_REQUESTS : "submits"

    JOB_CATEGORIES {
        bigint id PK
        varchar name UK "e.g. Electrician, Plumber"
        boolean is_active "default true"
        timestamp created_at
        timestamp updated_at
    }

    EMPLOYEES {
        bigint id PK
        varchar full_name
        varchar kennitala "nullable"
        varchar phone
        varchar email UK
        varchar address
        varchar nationality
        varchar languages_spoken
        boolean driver_license
        boolean own_car
        varchar cv_path
        varchar photo_path "nullable"
        timestamp created_at
        timestamp updated_at
    }

    EMPLOYEE_REGISTRATIONS {
        bigint id PK
        bigint employee_id FK
        bigint job_category_id FK
        date available_from
        text work_experience
        text skills
        timestamp created_at
        timestamp updated_at
    }

    COMPANIES {
        bigint id PK
        varchar company_name
        varchar contact_person
        varchar email UK
        varchar phone
        timestamp created_at
        timestamp updated_at
    }

    COMPANY_REQUESTS {
        bigint id PK
        bigint company_id FK
        bigint job_category_id FK
        int workers_needed
        text job_description
        varchar work_location
        date start_date
        varchar duration
        text additional_notes "nullable"
        timestamp created_at
        timestamp updated_at
    }

    USERS {
        bigint id PK
        varchar name
        varchar email UK
        varchar password
        timestamp created_at
        timestamp updated_at
    }

Diagram rendered with Mermaid — requires internet access on first load (loaded from a CDN below).

Notes on the diagram

  • USERS is the admin login only (Laravel's default table) — no relationship to the other tables in V1. It's drawn standalone deliberately.
  • EMPLOYEE_REGISTRATIONS carries a composite unique constraint on (employee_id, job_category_id) — not something Mermaid draws natively, so noting it here: this is what makes "register as Electrician, then later as Plumber" produce two rows, while "register as Electrician twice" updates the one row instead of duplicating it.
  • Identity vs. activity, on both sides: EMPLOYEES→EMPLOYEE_REGISTRATIONS and COMPANIES→COMPANY_REQUESTS are the same pattern for the same reason — one stable row per real-world person/company (deduplicated by email, ready to become a login account in V2), and many rows underneath for the things that actually repeat over time (a trade applied under, a hiring need submitted).
  • Why not a table named jobs: Laravel reserves that name for its own queue system, so the lookup table is job_categories.
